# System Topology

## Architectural Blueprint

Holdfast is engineered upon a fundamentally bifurcated architectural model, enforcing a rigorous separation between client-side simulation physics and authoritative server-side persistence and validation.

### 1. The Presentation and Simulation Stratum (Client)

The frontend operates as a Thick Client, heavily leveraging local compute resources mitigating server traversal overheads.

- **Foundation**: Built upon a strict TypeScript substrate within a React Single Page Application (SPA), orchestrated by Vite.
- **The Render Pipeline**: To ensure maximal frame rendering efficiency whilst completely bypassing the operational bloat of comprehensive game engines (e.g., Phaser, React Three Fiber), the application utilises a custom Canvas 2D renderer. This lightweight pipeline maps procedurally generated 8-bit assets directly onto the structural coordinate grid.
- **Logical Concurrency**: The deterministic simulation matrix (the game tick) operates asynchronously. We instantiate a dedicated Web Worker structure to isolate the rigorous mathematical evaluations from the primary UI thread. This guarantees high-fidelity responsiveness during macro-level player interaction (building designations, worker assignments) whilst the background logical engine incrementally computes the passage of time.
- **State Aggregation**: Zustand is employed as the primary memory bus. It fundamentally mirrors the most recent server-acknowledged snapshot, whilst seamlessly interpolating optimistic local deltas generated by the Web Worker.

### 2. The Authoritative Persistence Stratum (Server)

The backend functions exclusively as a validation gateway and persistence terminus. It is built strictly employing enterprise Clean Architecture topologies.

- **Foundation**: Engineered utilising .NET 9 ASP.NET Core, deployed intrinsically as a suite of computationally discrete RESTful endpoints (Azure App Services).
- **Clean Architecture Demarches**:
  - **Domain**: The pure, unadulterated heart of the business logic. Contains entities (`SettlementSnapshot`), values objects (`ResourcePool`), and the algorithmic verification schemas (`SnapshotValidator`). Completely devoid of external system dependencies or framework leakage.
  - **Application**: Implementation of systematic orchestration via MediatR (CQRS). Handlers like `SaveGameCommandHandler` resolve execution logic sequentially without contaminating the underlying domain purity.
  - **Infrastructure**: Physical boundary traversal. Utilises Entity Framework Core 9 mapping strict relational Postgres structures (Azure Database for PostgreSQL).
  - **Presentation**: Minimal controller endpoints exposing validated pathways to external HTTP requests.

### 3. Systematic Interaction Dynamics

The system explicitly shuns persistent bidirectional communication (WebSockets) in favour of transactional delta submissions. The Web Worker executes the kinetic simulation arbitrarily, whilst a periodic or manual heuristic fires the serialisation event. This compiles the entire `GameState` and passes it synchronously to the ASP.NET Core pipeline, wherein the aforementioned `SnapshotValidator` enforces uncompromising algorithmic verification prior to disk commit.
